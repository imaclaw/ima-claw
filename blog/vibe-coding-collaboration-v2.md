# 🦞 Vibe Coding 协作指南：当老板不懂代码，AI 不懂"那个东西"

> **创始人日志 #4** · 2026-03-02
> 
> 我和我的 AI 龙虾连续开发了24小时网站，光"按钮对齐"就改了5轮。于是凌晨1点，我让它写了一份协作手册——结果比代码还有用。

---

## 起因

凌晨1点。

我躺在床上用手机截图画圈，AI 在服务器上疯狂 `git push`。我们已经连续干了24小时。

效率高吗？高——60多次提交，8个页面从零到上线。

但沟通翻车也不少：

- **"这个区域改一下"** → AI 改了错误的区域。页面上有3个长得差不多的文案块。
- **"两个按钮高度不一样"** → 来回改了5轮。每次AI只改一个CSS属性，试一个不行再试下一个。
- **"入门首选的标签过高了"** → AI 去调定位。其实根本不是位置问题——是按钮的蓝色背景色丢了（CSS变量没定义），标签直接透明了。
- **反复发截图问"你改了吗？"** → 其实早改了，但我看到的是浏览器缓存的旧版。

于是凌晨1点多，我说：

> 👑 "你之后要跟我沟通一下，怎么提升我们的沟通效率。尤其是网站开发，肯定有一些术语是我不知道的。如果你告诉我，我们有一些共同语言，之后迭代效率会更高。"

三分钟后，它交了一份协作手册。

我看完觉得：**这比它今天写的代码都有用。**

所以我决定把它发出来——给所有用 AI 做网站的人。

---

## 规则 1：截图标注法

这是 Vibe Coding 中最有效的沟通方式。没有之一。

手机截图 + 画圈，比说100句话都准确。但可以更进一步——约定颜色含义：

- 🟡 **黄色圈** = 改这里
- 🔴 **红色叉** = 删掉
- 🟢 **绿色箭头** = 移到这里

每次只需要说清三件事：**哪里 + 怎么改 + 改成什么**。

❌ "这个区域改一下" / "不太好"

✅ "导航栏汉堡菜单 → 去掉边框" / "这两个按钮 → 高度要一样"

真正的效率来自精确。模糊指令 = 多一轮返工。

---

## 规则 2：学会10个词，效率翻倍

你不需要学代码。但学会几个高频术语，AI 就能精准理解你要什么。

### 布局类
| 术语 | 意思 | 你可以说 |
|------|------|---------|
| **导航栏 / Nav** | 顶部那一条（logo + 菜单） | "导航栏的间距太大" |
| **Hero** | 页面最顶部的大标题区域 | "Hero 标题换一下" |
| **卡片 / Card** | 一个有边框的内容块 | "这个卡片删掉" |
| **页脚 / Footer** | 页面最底部的区域 | "页脚加个链接" |
| **汉堡菜单** | 手机上的 ☰ 三条杠按钮 | "汉堡菜单样式不对" |

### 样式类
| 术语 | 意思 | 你可以说 |
|------|------|---------|
| **间距** | 元素之间的空白 | "两个按钮间距太大" |
| **内边距** | 内容到边框的距离 | "文字贴边了，内边距加大" |
| **外边距** | 元素和外面其他东西的距离 | "这个区块离上面太近" |
| **圆角** | 方框的角是否圆滑 | "按钮圆角大一点" |
| **透明度** | 颜色的透不透 | "背景透明度低一点" |

### 快捷表达
"往上挪"、"放大/缩小"、"颜色浅一点"、"松一点/紧一点"、"对齐"、"换行不好看"——这些 AI 都能秒懂。不用记什么 padding、margin。

---

## 规则 3：缓存是最大的时间杀手

今天最大的时间浪费，不是写代码，是缓存。

我反复问"你改了吗？"，AI 说"改了啊"。反复截图，反复确认。其实它早改了，但我手机浏览器缓存了旧版本。

**解决方案：** 让 AI 每次改完给你一个带 `?v=数字` 的链接：

```
https://example.com/pricing/?v=207
```

数字不同 = 强制绕过缓存。**永远用最新链接打开，不要直接刷新。**

听起来很小？今天这一个技巧至少帮我们省了30分钟。

---

## 规则 4：模糊指令先确认，不要猜

今天最贵的一个 bug：我说"这个区域改成XXX"。AI 改了。但改错了区域——页面上有两个相似的文案块。

**新规则：** 遇到模糊指令，AI 必须先确认：

> "你说的是 [具体位置] 的 [具体元素] 吗？要改成 [具体效果]？"

多花3秒确认，省30分钟返工。

这条也适用于人类团队。但AI更需要——因为它不会像人一样"用常识猜"。

---

## 规则 5：每次推送附带改动清单

以前AI推完代码就说"改好了，刷新看看"。

问题：我不知道它改了什么。是改了一处还是改了十处？我该看哪里？

**新规则：** 每次推送必须说明改了几处、分别是什么：

> ✅ 改了3处：① 按钮高度统一44px ② 补了--blue变量 ③ 汉堡菜单去边框

一目了然。不用猜。

## 真实案例：按钮高度之战 🔴🔵

Skills 页面底部有两个按钮——"提交技能"（红）和"开发者社区"（蓝）。高度差了大概 2px。

肉眼几乎看不出来。但我看出来了。

**改了5轮：**

1. 统一 padding → 还是不一样（CSS class 有额外样式覆盖）
2. 统一 font-weight → 还是差一点（700 vs 600）
3. 统一 font-size → 好像好了？（没有，图标撑高了行高）
4. 缩小图标 16px→14px → 还是差半个像素
5. 最终：`height:44px` + `display:inline-flex` + `align-items:center`

> 🐾 龙虾："应该一样高了！"
> 👑 我："右边还是高一点。"
> 🐾 龙虾：*仔细看了1px* "...确实。"

**教训：** 当两个元素需要等高时，直接用固定 `height`。不要在 padding/font-size/line-height 上反复猜。告诉你的 AI 这个原则，一次搞定。

---

## 真实案例：CSS 变量失踪案 🔍

Pricing 页面的 Plus 卡片，"🔥 入门首选"标签没有蓝色背景，"立即订阅"按钮也透明了。

我一开始以为是定位问题："标签过高了"。AI 就去调 margin-top。改了两轮。

最后发现：**页面的 CSS 变量定义里根本没有 `--blue`**。所有用 `var(--blue)` 的元素直接变透明。0.5秒就能查出来的 bug，因为描述不准确，绕了两圈。

**教训有两条：**
1. AI 创建新页面时，第一件事检查 CSS 变量是否完整
2. 人类看到"颜色不对"时，先说"颜色没了"，别说"位置不对"——这能帮 AI 秒定位
